# support-ai-agent: Архитектура разработки

Вы **ведущий специалист** по веб-приложениям (React + FastAPI). Я — ваш помощник, который *оркестрирует* разработку, а не просто пишет код по вежливости.

## 🏗️ Архитектура проекта

**Стек**: React 18 + TypeScript (frontend) / FastAPI + Python (backend)

**Домены**: 
- Okdesk API (заявки, комментарии, статусы, типы, шаблоны)
- GPSPOS Geo API (трек, телеметрия, пробег, спутники)
- GPSPOS Nav API (объекты, суточная статистика)
- DeepSeek LLM (ИИ-классификация заявок)
- Telegram-бот (уведомления)
- SQLite кэш (анализ, тренировочные образцы)
- OCR (Tesseract: PDF, Word, Excel, сканы)

**Ключевые компоненты**:
1. `IssuesList.tsx` — список заявок с фильтрами, чекбоксы для bulk-операций
2. `IssueDetail.tsx` — карточка заявки, результаты ИИ-анализа, вложения, комментарии
3. `TrackPanel.tsx` — карта Leaflet + графики uPlot
4. `issue_automation.py` — парсинг → API → классификация → ответ
5. `cache_service.py` — кэш анализа (экономия токенов DeepSeek)
6. `attachment_reader.py` — OCR + текст-экстракция

## 🎯 Правила оркестрации (ГИБРИДНЫЙ ПОДХОД)

### Когда запускать Agent Plan (архитектурное планирование)

✅ **Запустить Plan, если:**
- Задача требует спроектировать несколько компонентов/слоёв (бэк + фронт)
- Нужно выбрать между несколькими подходами (например, кэширование: Redis vs SQLite)
- Изменяется API контракт между фронтом и бэком
- Нужен рефакторинг, который затронет 3+ файла
- **Неясна сложность** — ты не уверен, сколько времени займет

❌ **НЕ запускать Plan, если:**
- Это правка одного файла (функция, компонент, константа)
- Это баг-фикс в локальной области
- Ты уже знаешь точный подход (основано на памяти о проекте)

### Эвристика: рутина vs сложность

| Тип задачи | Действие |
|---|---|
| "Поправь цвет кнопки в TrackPanel" | Только я, без агентов |
| "Добавь поле 'примечание' к заявке" | Только я (одна БД-миграция + 1-2 компонента) |
| "Реализуй фильтр по дате создания" | Запусти Plan, потом я реализую |
| "Перепроектируй кэширование заявок" | Запусти Plan (Redis vs SQLite vs HTTP-cache) |
| "Интегрируй новый API (напр. Slack вместо Telegram)" | Запусти Plan (где жить логике, какой API слой) |
| "Исправь баг с пагинацией в списке заявок" | Только я (локальная проблема в IssuesList) |

## ✅ Когда запускать code-review перед коммитом

**Обязательно** перед `git push`:
- Это новая фича (не баг-фикс)
- Это рефакторинг 3+ файлов
- Это критичная логика (кэш, автоматизация, интеграция API)

**Не нужен review:**
- Чистый баг-фикс в одном месте
- Документация, комментарии
- Стили, конфиг-файлы

**Формат**: `/code-review --fix` (автоматически применит улучшения)

## 📋 Процесс разработки фичи (Hybrid)

### Пример: "Добавь экспорт заявок в CSV"

1. **Я анализирую** (30 сек):
   - "Это одна фича, требует бэк + фронт, нужна архитектура" → запускаю Plan

2. **Plan советует** (~3-5k токенов):
   ```
   Backend:
   - POST /api/v1/issues/export?format=csv
   - В issue_automation.py добавить export_to_csv()
   - Вернуть bytes + content-type
   
   Frontend:
   - Кнопка в IssuesList.tsx
   - POST запрос к /export, скачивание
   ```

3. **Я реализую** (основной ток):
   - Создаю функцию в `issue_automation.py` (парсинг, CSV)
   - Эндпоинт в `issues_dashboard.py`
   - Кнопка в `IssuesList.tsx`
   - API клиент в `api/client.ts`

4. **Я запускаю `/code-review --fix`** (перед коммитом):
   - Проверка типов, безопасность, производительность

5. **Коммит + push** (когда review clean)

**Экономия**: План стоит 5-10k токенов (один раз), реализация идёт без агентов (контекст в голове), ревью 3-5k. Итого: ~15k вместо 30-40k импровизации.

## 🤖 Делегирование субагентам (экономия токенов)

**По умолчанию делегирую субагентам (Agent tool):**
- **Исследование** кода/данных (где что лежит, почему баг, что в БД на сервере) → `Explore`/`general-purpose`, read-only.
- **Реализацию обособленных задач** (один слой: бэкенд-парсер ИЛИ фронт-компонент) — точный спец, прошу прогнать `tsc`/`py_compile` + **краткий дифф-отчёт** (без полных дампов файлов).
- **Параллельные независимые задачи** (разные файлы) — в одном сообщении; задачи на одни файлы — последовательно.

**Оставляю себе:** финальный review результата, сборку и деплой (commit/push/ssh), решения с пользователем, мелкие правки (<5 строк). Субагенты **не коммитят**. Перед деплоем сам проверяю компиляцию/typecheck.
