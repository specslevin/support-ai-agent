## Автоматизация (Phase A) + анализ

```
app/services/issue_automation.py        # IssueAutomationService: parse → telemetry → classify → draft → track → training sample
app/services/attachment_reader.py       # PDF/Word/Excel + OCR (PyMuPDF+tesseract rus+eng)
app/core/gpspos_geo/service.py          # find_object_by_plate, get_daily_stats, get_packets
```
- **Пробег**: считается из реального трека (сумма haversine по пакетам, без телепортов) — НЕ из DailyStat.length (он отстаёт, не отражает догрузку из чёрного ящика).
- **Парсинг**: гос.номер (с/без региона), дата неисправности (приоритет «за <дата>», не первая дата отчёта), ПЛ/одометр (требует единицу км/м).
- **Категории**: Данные верны / Глушение / Не было питания / Терминал подключился / Изменили настройки / Диагностика. Глушение — по телепортам трека/потере спутников (НЕ по одиночным выбросам скорости). Обрыв трека → «Терминал подключился» (потеря связи + поздняя выгрузка). Занижение пробега vs ПЛ → conf ≤0.7 + needs_review.
- Текст вложений (вкл. OCR сканов) идёт в парсинг + промпт DeepSeek.
- Каждое решение через систему → строка в `training_samples` (факты телеметрии + ответ оператора) для будущего обучения.
- ⚠️ Снапшот в моменте ≠ финальные данные (поздняя выгрузка) — поэтому неопределённость честно помечается на проверку.

**Дата неисправности** (приоритет): «Дата неисправности <дата>» → «в системе [с] <дата>» → «за <дата>» → дата в теме → первая дата. (Первая дата в тексте обычно дата отчёта/отправки, не неисправности — особенно акты Волжского ПО.)
**Поиск объекта geo** по ядру гос.номера без региона (Х371РХ64 → находит «Х371РХ ГАЗ»).
**Назначение ответственного**: `PATCH issues/{id}/assignees` (мн.ч.) + `{assignee_id, group_id}` — группа сотрудника ОБЯЗАТЕЛЬНА (бэкенд резолвит её из employees/list). Путь `issues/{id}/assignee` (ед.ч.) — 404.
**OCR под systemd**: PATH сервиса = только venv, без `/usr/bin` → pytesseract не находил бинарь и OCR молча не работал. Путь к tesseract прописан явно в `attachment_reader._ensure_tesseract_cmd`.
**Гос.номер латиница↔кириллица**: канонизация (A759PC → А759РС) в `_norm_plate`.
**Спецтехника**: формат 4 цифры + 2 буквы в любом порядке (5297СУ / СУ5297); сопоставление с geo по канонической сигнатуре (`_special_sig`), т.к. порядок цифр/букв и регион в данных geo непостоянны.
**Дата неисправности** также из ISO (YYYY-MM-DD) — для дочерних заявок (раньше ISO не читался; теперь дочерние пишут «Дата неисправности: ДД.ММ.ГГГГ»).
**Разбор по объектам**: глушение приоритетнее совпадения пробега (спуфинг/круг = трек недостоверен, не «данные верны» даже если пробег совпал). У каждого объекта — кнопка 🗺 (трек этого ТС через `track?plate=&date=`).
**Разбор «общих» заявок**: заявка с ≥2 извлекаемыми вложениями (по акту на ТС) → одиночный автоанализ скрыт, показывается «Разбор по объектам» (таблица вердиктов). По объектам «данные верны» — кнопка создания вложенных заявок.
**Создание заявок в Okdesk**: `POST issues` тело `{"issue": {...}}`. Тип на создании ИГНОРИРУЕТСЯ (ставится отдельным `PATCH issues/{id}/types`). Тип «Расхождение пробега» требует кастомный параметр `address` (Местоположение техники) непустым; коды параметров: `address`, `contact_person`, `tel_person`.
**Разбор по форматам вложений** (`automate_batch`): `_parse_act_blocks` (мульти-акт PDF, один акт = один ТС, своя дата/ПЛ), `_parse_body_vehicles` (список ТС в теле), `_parse_summary_table` (сводная таблица), `_parse_grouping_table` (XLSX «Группировка <дата>»). Глобальный дедуп ТС по вложениям.
**Формат-профили ПО** (`_FORMAT_PROFILES`/`resolve_format_profile`): мягкая привязка по подстроке `company_name` (Волжское/Жигулёвское/Чапаевское/Самарское/Ульяновское ПО и др. дочерние Россети Волга) задаёт стратегию разбора (тема/вложения, single/multi, формат акта) и подсказку для промпта.
**Спецтехника** (`_is_special_vehicle`): спецномер (4 цифры + 2 буквы в любом порядке) ИЛИ модель из списка → без км-вердикта; в таблице разбора бейдж «спецтехника», в анализе предупреждение.
**Детектор «ложного wait»** (`_client_awaiting_remote_diag`): клиент подтвердил питание/массу/индикацию ИЛИ просит «удалённую диагностику» → шаблон «включите питание» исчерпан, мяч на нашей стороне → предупреждение «удалённая диагностика».
**Пробег от клиента**: «по путевому листу/одометру» (`_WAYBILL_KM_RE`) и «в системе ГЛОНАСС/ССМ/СМС» (`_GLONASS_KM_RE`) — заявленное клиентом ОТДЕЛЬНО от реальной телеметрии geo; колонка «ГЛОНАСС заявл.» рядом с «По факту». Date-guard: ГЛОНАСС/пробег не путать с датой (`17.06.2026`) и телефоном.
**Вердикт-по-дате**: ключ строки разбора = (номер, дата, файл) — у одного ТС за разные даты разные вердикты (63617). Допустимые вердикты: Глушение / Данные верны / Не было питания / Нет данных / Терминал подключился / Проверить.
**Статусы смены** (`resolve`/bulk): ALLOWED = completed/delayed/opened/closed/wait/no_time/inst_fin. `wait`=«В работе» (без обязательного ответа клиенту), `no_time`/`delayed` требуют `delay_to`. Кнопка «В работе» снизу карточки (баг 64453/64306 исправлен).
**Калибровка глушения** (`_derive_flags`): пороги откалиброваны на 159 «глушений» vs 58 «не-глушений» из training_samples (recall 73%→78%, ложные ~10%). Спутниковые/нулевые сигналы = глушение только если ТС двигалось (у стоящего мало спутников — норма, 64275). Отдельная ветка «стоячее глушение из-за подавления» (64657): `low_sat>0.9` + питание в норме → глушение даже без признаков движения (само подавление обнулило скорость/трек). «Не было питания» НЕ ставим движущемуся ТС (64281) — только стоящему с низким напряжением.
**Петля обратной связи ИИ** (`ai_feedback`): оператор оценивает разбор — good (верно)/bad (+тип ошибки wrong_verdict/wrong_plate/wrong_date/wrong_mileage/other + комментарий + правильная категория). Хранится с категорией, которую выдал ИИ. Экран «Оценки ИИ» (`AiFeedbackReview.tsx`) — основа для роста few-shot из верных разборов и точечной правки промпта/правил по ошибкам.
**Копирование монтажнику** (`installer_export`): два готовых текста — КАЛЕНДАРЬ (телефон/компания/ТС/«не в сети с …») и МЕССЕНДЖЕР (объект/местоположение/контакт/телефон). Телефон и контакт берутся из ПАРАМЕТРОВ заявки (`tel_person`/`contact_person`), фолбэк телефона — `get_contact` (64239). Недостающие поля = плейсхолдер `____`, запрос не падает; read-only (доступно demo).
**Редактирование параметров заявки** (`POST /parameters`): Местоположение техники/Контактное лицо/Номер телефона → `PATCH issues/{id}` с `custom_parameters` (64197). Заполняет обязательные поля, без которых Okdesk не пускает в статус «В работе». UI-блок `EditableParameters` в карточке.
**Личные шаблоны** (`app_templates.user_id`): NULL = общий (виден всем), username = личный (виден/правится только владельцу). Дедуп и фильтр видимости scoped по владельцу; миграция добавляет колонку (`database._column_exists`).
**Модель ИИ**: `deepseek-v4-flash` (env `DEEPSEEK_MODEL`, default в `core/ai/llm.py`); ~$0.20/мес. Исправлено с ошибочного `deepseek-chat`.
**Шаблоны (апгрейд)**: сортировка по `usage_count`, дедуп по контенту, плоский список с фильтром, инкремент usage при вставке, динамические плейсхолдеры `[вчера]`/`[завтра]`/`[сегодня]`. Шаблоны категории подмешиваются в few-shot к `automate` (`fetch_templates_for_category`).
**Извлечение сводной таблицы** (`_parse_summary_table`, 63317 Саратовские РС): «Автомобиль г/н | Дата выезда | Пробег по ГЛОНАСС | Пробег по 1С» → 4-кортеж (номер, дата, glonass=заявл. система, sheet=ПЛ/1С). Интервал дат «DD-DD.MM», пропуск пустых вложений.
**Альтернативный номер** (`_resolve_alt_plate`, object_not_found): если объект по номеру из темы не найден в geo, ИИ переключается на валидный гос.номер из тела/вложений.
