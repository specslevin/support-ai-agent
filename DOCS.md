## Что это

Центр управления заявками технической поддержки GPS-мониторинга (GPSPOS).  
Интегрирован с Okdesk (helpdesk), GPSPOS Geo API, Telegram-ботом и DeepSeek LLM.

## Функциональность dashboard

### Список заявок
- Фильтры: № заявки (external_id), тема, статус, **ответственный** (+ «Не назначен»), компания — **комбинируются** (AND)
- Чекбоксы + массовое редактирование (ответственный/тип/статус с шаблонами)
- Сортировка по дате/приоритету, пагинация (размер запоминается)
- Обновление кэша вручную
- **Личные сохранённые фильтры** («Мои фильтры»): CRUD, сортировка, подсветка активного (кнопка + строка списка), кнопка «Сбросить» после всех фильтров

### Карточка заявки (IssueDetail)
- Live данные из Okdesk (не кэш): тип, описание, сроки, автор, параметры
- **Смена типа заявки** — inline выпадашка, доступные типы берутся из Okdesk API
- **Смена статуса** — клик на бейдж статуса → dropdown с доступными переходами (цвета Okdesk)
  - ALLOWED на бэке (`resolve`/bulk): `completed`/`delayed`/`opened`/`closed`/`wait`/`no_time`/`inst_fin` (баг 64453/64306 — раньше wait/no_time давали HTTP 400)
  - Бизнес-правила фронта: «В работу» только для типов `departure`/`departure_fuel`; финальные статусы заблокированы если тип `inner`
  - Форма с комментарием (обязательный для delayed/no_time) + поле «Отложить до» / `delay_to` (обязателен для delayed/no_time)
  - Кнопка **«В работе»** (wait) снизу карточки — смена статуса без обязательного ответа клиенту
  - Чекбокс «Публичный комментарий» (по умолчанию включён)
- **Добавить комментарий** — textarea + выбор шаблона + чекбокс публичности
- **Системные уведомления Okdesk** отображаются серым (детект по автору)
- **Шаблоны ответов** — из okdesk-console (отдельный сервис на порту 8000), категории с цветами, избранные, поиск
- **Назначить ответственного** — dropdown сотрудников
- **Быстрое решение** — кнопки «Решить»/«Ожидание» прямо в поле комментария (без модалок), шаблоны
- **Вложения** — список с иконками/размером, бейдж «🤖 ИИ читает», просмотр/скачивание

### ИИ-автоанализ «Расхождение пробега»
- Кнопка «🤖 Автоанализ»: парсинг заявки (гос.номер, дата неисправности по «за <дата>», ПЛ/одометр с единицей) → данные geo → классификация → черновик ответа.
- **Пробег** считается из реального трека (haversine по пакетам, без телепортов), а не из устаревшего `DailyStat.length` — отражает догрузку из чёрного ящика.
- **Категории**: Данные верны / Глушение (по телепортам трека и потере спутников, не по выбросам скорости) / Не было питания / Терминал подключился (обрыв связи + поздняя выгрузка) / Изменили настройки / Диагностика.
- Текст вложений (включая OCR сканов) идёт в парсинг и в промпт DeepSeek.
- Занижение пробега vs ПЛ → уверенность ≤0.7 + флаг «нужна проверка».
- ИИ описывает **хронологию событий** «подключился → выгрузил данные → глушение».
- **Ручная правка гос.номера/даты** и в одиночном ИИ-анализе (не только в разборе по объектам): пометка неверного ФОРМАТА гос.номера, трек-панель следует за исправленным номером.
- Кнопки внизу блока: «↓ В комментарий» / «✓ Ответить и решить».

### Карта трека + графики (TrackPanel)
- Кнопка «🗺 Карта и графики» в шапке деталей → панель выезжает влево; свернуть — кнопкой «◀ Свернуть» в самой панели.
- **Статус объекта**: онлайн/офлайн, дата последнего сообщения, IMEI и телефон терминала (копирование по клику).
- Карта (Leaflet + OSM): трек, телепорты подсвечены пунктиром, старт/финиш, курсор-маркер при наведении на график.
- Графики (uPlot): скорость/напряжение/спутники, зум колёсиком, drag-выделение.
- Статистика по выбранному интервалу (пробег, скорости, спутники, питание, длительность).

### Массовое редактирование
- Чекбоксы (нейтральный серый) в строках + «выбрать всё» на странице.
- Панель действий: ответственный / тип / статус (с комментарием и шаблонами) для всех выбранных.

### Разбор «общих» заявок (batch)
- Заявка с ≥2 ТС (несколько вложений ИЛИ список в теле) → одиночный автоанализ скрыт, кнопка «🗂 Разбор по объектам (N)».
- **Парсинг по форматам вложений**: мульти-акт PDF (`_parse_act_blocks`, один акт = один ТС со своей датой/ПЛ), список ТС в теле (`_parse_body_vehicles`), сводная таблица (`_parse_summary_table`), XLSX «Группировка» (`_parse_grouping_table`). Глобальный дедуп ТС.
- **Формат-профили ПО** (`_FORMAT_PROFILES`): по `company_name` (дочерние Россети Волга) выбирается стратегия разбора и подсказка для промпта.
- Таблица: гос.номер / дата / ПЛ / **ГЛОНАСС заявл.** (от клиента) / **По факту** (реальный трек) / вердикт + 🗺 трек каждого ТС. Спецтехника помечается бейджем (без км-вердикта). Акты/строки без распознанного номера видимы в таблице (не теряются).
- **Ручная правка гос.номера/даты** в строках разбора: пометка неверного формата, трек-панель следует за исправленным номером.
- **Улучшен разбор кейсов с ошибками** (по оценкам ИИ): мульти-акт PDF, даты листов, номер из темы/комментариев, fuzzy-гео сопоставление.
- **Редактируемый вердикт** (`POST /batch/verdict`), ключ строки = (номер, дата, файл) — один ТС за разные даты = разные вердикты. Вердикты: Глушение / Данные верны / Не было питания / Нет данных / Терминал подключился / Проверить.
- Глушение приоритетнее совпадения пробега: спуфинг (круг/телепорты) = трек недостоверен → не «данные верны», даже если пробег совпал.
- **Один общий ответ** (`compose_answer`): группировка ТС по (отредактированным) вердиктам — для сводных/ОДКР заявок.
- По объектам «данные верны»/«нет данных» — **создание вложенных заявок** (`create_children`): дочерняя заявка в Okdesk с `parent_id`, тип «Расхождение пробега».

### Кэш анализа
- Результаты автоанализа и разбора сохраняются в `result_cache` (kind=automate|batch).
- UI показывает сохранённый результат без повторного прогона ИИ (экономия токенов); кнопка «Обновить» = форс-прогон (POST). GET-эндпоинты отдают кэш.

### Чтение вложений / OCR
- OCR **не блокирует сервер** (устранён баг CPU 100% и цикл «Ошибка разбора»).
- **Накопительный возобновляемый OCR** больших многоактных PDF с кэшем `ocr:<att_id>`; **авто-дораспознавание** больших PDF на фронте.

### Распознавание гос.номеров
- Обычные: А123ВС[64], с/без региона, латиница↔кириллица (A759PC→А759РС).
- Улучшено: **префикс-склейка** («г.н.К862ВТ»→«г.н. К862ВТ»), дизамбигуация без региона, транслит латиница→кириллица.
- Спецтехника: 4 цифры + 2 буквы в любом порядке (5297СУ / СУ5297); сопоставление с geo по канонической сигнатуре.
- Связанные заявки (родительская/дочерние) кликабельны — открываются по № в панели деталей.

### Создание заявок в Okdesk (нюансы)
- `POST issues` тело `{"issue": {...}}`. Тип на создании ИГНОРИРУЕТСЯ → ставится отдельным `PATCH issues/{id}/types`.
- Тип «Расхождение пробега» требует непустой кастомный параметр `address` (Местоположение техники). Коды: `address`, `contact_person`, `tel_person`.

### Обучение ИИ (задел)
- Каждое решение через систему → строка в `training_samples` (телеметрия + ответ оратора + статус) для будущего few-shot/fine-tune.
- **Few-shot retrieval** подключён к `automate`: `find_similar_resolved` (похожие решённые кейсы по категории/номеру/флагам/отправителю) + шаблоны той же категории (`fetch_templates_for_category`) подмешиваются в промпт как примеры.

### Петля обратной связи ИИ (ai_feedback)
- Оператор оценивает качество ИИ-разбора: **good** (верно) / **bad** (+ тип ошибки `wrong_verdict`/`wrong_plate`/`wrong_date`/`wrong_mileage`/`other` + комментарий + правильная категория).
- Хранится в таблице `ai_feedback` (модель `AiFeedback`) вместе с категорией, которую выдал ИИ, и автором оценки.
- Эндпоинты: `POST /issues/{id}/ai_feedback`, `GET /issues/{id}/ai_feedback` (последняя), `GET /issues/ai_feedback/list[?rating=]`. Экран «Оценки ИИ» (`AiFeedbackReview.tsx`) — хорошо разобрано / с ошибками; основа для роста few-shot и точечной правки промпта/правил. Отметка **«исправлено»** для проработанных оценок с ошибками.

### Копирование монтажнику (installer_export)
- `GET /issues/{id}/installer_export` — два готовых текста для копирования: **КАЛЕНДАРЬ** (телефон / компания / ТС / «не в сети с ДАТА») и **МЕССЕНДЖЕР** (объект / местоположение / контакт / телефон).
- Телефон и контакт берутся из **параметров заявки** (`tel_person`/`contact_person`), фолбэк телефона — `get_contact` (64239). Недостающее поле → плейсхолдер `____`, запрос не падает. Read-only (доступно demo).

### Редактирование параметров заявки (parameters)
- `POST /issues/{id}/parameters` — обновить Местоположение техники / Контактное лицо / Номер телефона (`address`/`contact_person`/`tel_person`) через `PATCH issues/{id}` с `custom_parameters` (64197). Шлются только переданные поля.
- Заполняет обязательные поля, без которых Okdesk не пускает заявку в статус «В работе». UI-блок `EditableParameters` в карточке. Доступно только не-demo.

### Личные шаблоны (user_id)
- Колонка `app_templates.user_id`: `NULL` = общий шаблон (виден всем), `username` = личный (виден/правится только владельцу). Миграция добавляет колонку (`database._column_exists`).
- Дедуп и фильтр видимости scoped по владельцу. Сортировка по `usage_count`, инкремент при вставке, динамические плейсхолдеры `[вчера]`/`[завтра]`/`[сегодня]`.

### Калибровка детектора глушения (`_derive_flags`)
- Пороги откалиброваны на 159 «глушений» vs 58 «не-глушений» из `training_samples` (recall 73%→78%, ложные ~10%).
- Спутниковые/нулевые сигналы = глушение только если ТС двигалось (у стоящего мало спутников — норма, 64275). Отдельная ветка «**стоячее глушение из-за подавления**» (64657): `low_sat>0.9` + питание в норме → глушение даже без признаков движения (подавление само обнулило скорость/трек).
- «Не было питания» НЕ ставим движущемуся ТС (64281), только стоящему с устойчиво низким напряжением; в промпте запрещён ложный вывод «Не было питания».
- При `object_not_found` (объект по номеру из темы не найден в geo) ИИ переключается на валидный гос.номер из тела/вложений (`_resolve_alt_plate`).

### Управление пользователями (Блок 3)
- Admin **создаёт операторов** (управление пользователями в UI).
- Бейдж **«можно авто»** на операторах, которым доверена автоматизация.

### ИИ-чат
- Раздел «ИИ-чат» с бейджем **«скоро»** (в разработке).

### Авторизация
- **Fail-fast** при пустом/дефолтном `AUTH_SECRET` в production (сервис не стартует с небезопасным секретом).

### В работе (не завершено)
- **Детект нескольких ТС по ТЕЛУ письма** (не только по теме) — заявки, где тема = дата, а список ТС в теле (65649). Не решает случай спецномеров во вложении (65676).

| Метод | Endpoint | Назначение |
|---|---|---|

| PATCH | `issues/{id}` | **Редактировать параметры** — тело `{"issue": {"custom_parameters": {address, contact_person, tel_person}}}` (заполнить обязательные поля для «В работе») |

| GET | `employees/list` | Список сотрудников |
