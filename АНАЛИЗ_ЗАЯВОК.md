# Как работает ИИ-анализ заявок

Документ описывает, как устроен автоматический анализ заявок (в первую очередь
«Расхождение пробега»), откуда берутся данные, как принимается решение, какие у
системы слабые места и чем отличается анализ заявки **с типом «расхождение
пробега»** от анализа **без указания типа**.

Главный модуль: [`app/services/issue_automation.py`](app/services/issue_automation.py),
класс `IssueAutomationService`, метод `automate()`.

---

## 1. Источники данных

| Источник | Что даёт | Где в коде |
|---|---|---|
| Okdesk | тема, описание, комментарии, вложения заявки | `OkdeskService` |
| Вложения (PDF/Word/Excel/сканы) | акты, путевые листы → OCR-текст (возобновляемый накопительный OCR больших многоактных PDF, кэш `ocr:<att_id>`, вынесен в поток — не блокирует сервер, авто-дораспознавание) | `app/services/attachment_reader.py` |
| GPSPOS Geo API | объект по гос.номеру, сырые пакеты телеметрии, суточная статистика, статус (последний выход на связь) | `app/core/gpspos_geo/service.py` |
| DeepSeek LLM | классификация причины + формулировка ответа | `app/core/ai/llm.py` |
| SQLite (TrainingSample) | похожие решённые кейсы для few-shot | `cache_service.find_similar_resolved` |

## 2. Конвейер анализа (mileage-путь)

```
parse_issue → [fallback: _extract_with_llm] → gather_telemetry →
_heuristic_category (флаги) → _draft_with_llm (DeepSeek) → AutomationResult
```

### 2.1. Парсинг входа — `parse_issue()`
Из `тема + описание + текст вложений` регэкспами извлекаются:
- **Гос.номер** (`_PLATE_RE`, нормализуется в кириллицу: `Y538OK → У538ОК`).
  Распознавание усилено (сессия 14): **префикс-склейка** — «г.н.К862ВТ» отделяется
  в «г.н. К862ВТ», иначе lookbehind резал приклеенный номер; **транслит** латиница→
  кириллица (`M203YO → М203УО`, 64481); **дизамбигуация** номера без региона. Дедуп
  ТС ведётся по ключу без хвоста региона (`_plate_dedup_key`): «М469НА73» и «М469НА»
  считаются одним ТС;
- **Дата неисправности** — каскад приоритетов: явные маркеры («дата неисправности»,
  «в системе», «за …»), затем дата в теме, затем первая дата в тексте. Маркер
  «период за» исключён, чтобы не цеплять дату периода проверки (сводные письма);
- **Пробег** по путевому листу и «в системе/в ПК».

- **Пробег по системе ГЛОНАСС/ССМ/СМС** (`_GLONASS_KM_RE`) — то, что КЛИЕНТ видит в
  своём ПК, отдельно от РЕАЛЬНОЙ телеметрии geo. В таблице разбора показывается
  колонкой «ГЛОНАСС заявл.» рядом с «По факту» (реальный пробег по треку).
- **Date-guard**: ГЛОНАСС/пробег не путаются с датой (`17.06.2026` не даёт «17.06»)
  и с телефоном ответственного (`_safe_km`: >5 цифр до точки → отброс).

Если регэкспы не нашли номер ИЛИ дату — вызывается `_extract_with_llm()` (LLM
достаёт поля из нестандартных формулировок). Токены тратятся только при неудаче
регэкспов.

**Альтернативный номер** (`_resolve_alt_plate`): если объект по номеру из темы не найден
в geo (`object_not_found`), система пытается переключиться на валидный гос.номер из тела/
вложений заявки — частая ситуация, когда в теме опечатка/чужой номер, а правильный есть в акте.

**Спецтехника** (`_is_special_vehicle`): спецномер (4 цифры + 2 буквы в любом порядке,
напр. `5297СУ`/`СУ5297`, либо регион+серия `64СН3847`) ИЛИ модель из списка. У такой
техники нет км-пробега → вердикт без сравнения пробега, в таблице бейдж «спецтехника».

**Детектор «ложного wait»** (`_client_awaiting_remote_diag`): если в комментариях
клиент подтвердил питание/массу/индикацию ИЛИ просит «удалённую диагностику», значит
стандартный шаблон «включите питание» исчерпан — мяч на нашей стороне, заявка зависает.
В анализе появляется предупреждение «удалённая диагностика».

### 2.2. Телеметрия — `gather_telemetry()`
По гос.номеру ищется объект в GPSPOS (`find_object_by_plate`), строится **окно
суток в МСК** (важно: сервер в UTC, трекеры в МСК (+3) — окна строятся хелперами
`_msk_day_window_ms`), и запрашиваются:
- суточная статистика (`DailyStat`);
- **сырые пакеты** (`get_packets`) — реальный пробег считается по ним (гаверсинус
  между точками), т.к. `DailyStat.length` может отставать (поздняя выгрузка из
  чёрного ящика);
- если данных за дату **нет** — дополнительно берётся **дата последнего выхода
  объекта на связь** (`get_object_status().time`).

### 2.3. Эвристика — `_derive_flags()` + `_heuristic_category()`
Детерминированно выставляются флаги и выбирается «подсказка» категории:
- `power_off` — доля пакетов с низким напряжением > 20% **и ТС не двигалось**
  (движущийся терминал запитан — низкие near-zero пакеты не считаем за отключение, 64281);
- `jamming` — телепорты трека / спутниковые потери **в движении** (у стоящего ТС
  мало спутников — это норма, не глушение, 64275); телепорты со spoof-скоростью —
  признак глушения независимо от движения. Отдельная ветка **«стоячее глушение
  из-за подавления»** (64657): `low_sat>0.9` при идущих пакетах и питании в норме —
  глушение даже без признаков движения (подавление само обнулило скорость/трек).
  Пороги откалиброваны на 159 «глушений» vs 58 «не-глушений» (recall 73%→78%, ложные ~10%);
- `track_gap` — большой разрыв трека;
- `no_data` — пакетов нет.

Порядок: нет данных → «Диагностика»; нет питания → «Не было питания»; глушение →
«Глушение»; пробег сходится → «Данные верны»; разрыв трека → «Терминал
подключился».

**Хронология событий** (сессия 14): когда факты складываются в последовательность,
ИИ описывает её по порядку — «терминал подключился → выгрузил накопленные данные →
глушение» (кейс 64915), а не выдаёт разрозненные флаги.

### 2.4. Детерминированная ветка «нет данных за дату»
Когда `no_data`, в факты добавляется `ситуация_с_данными`:
- последнее сообщение **раньше** даты неисправности → объект давно молчит →
  **удалённая проверка силами нашей техподдержки** (категория «Диагностика»);
- данные появились **после** даты (`данные_возобновились=true`) → «в указанную
  дату не было питания, восстановлено <дата>, трек за период не вернуть»;
- последнее сообщение позже, но данных нет → выезд бригады.

### 2.5. LLM — `_draft_with_llm()`
В DeepSeek (`deepseek-v4-flash`, temperature 0.7) уходят: каталог из 6 категорий,
JSON-факты телеметрии, комментарии заявки, до 3 похожих решённых кейсов (few-shot)
и подсказка-эвристика. Модель возвращает строгий JSON
`{category, answer, confidence, reasoning}`. Промпт требует: гибкость в
формулировках **в рамках** категорий, без выдумывания данных, кратко.

### 2.6. Финал
Если уверенность < 0.85 или сильное занижение пробега → `needs_review`. Результат
кэшируется в `result_cache` по `(issue_id, "automate")`.

## 3. Разница: тип «расхождение пробега» vs без типа

В `automate()` есть развилка (`is_mileage = "пробег" in issue_type`):

### Mileage-путь (тип «расхождение пробега» ИЛИ распарсились номер+дата)
Полный конвейер из раздела 2: телеметрия → детерминированные флаги → DeepSeek с
**каталогом категорий пробега** и few-shot. Ответ структурирован (категория +
черновик), решения опираются на жёсткие пороги телеметрии. Это «рельсовый» режим:
предсказуемый, экономный по токенам, заточен под расхождение пробега.

### Общий путь без типа — `_draft_general()`
Включается, когда: **не распарсились** гос.номер/дата, ИЛИ тип заявки **явно не
«пробег»**, ИЛИ объект/телеметрия не нашлись. Это двухпроходный «контрольный
центр»:
1. **PASS 1** — один вызов LLM: определить намерение (intent) и какие данные нужны
   (объект, статус, суточная статистика);
2. **FETCH** — до 3 «дешёвых» обращений к Geo (объект → статус → суточная
   статистика);
3. **PASS 2** — один вызов LLM: составить ответ строго по собранным фактам.

Отличия от mileage-пути:
- **гибче, но менее структурировано** — нет каталога из 6 категорий пробега;
  категория = человекочитаемый ярлык намерения («Объект не на связи», «Запрос
  трека», «Настройки объекта» и т.п.);
- **всегда `needs_review=True`** — общий путь не закрывает заявку сам, оператор
  подтверждает;
- здесь тоже есть детерминированная подсказка: если последний выход на связь
  **раньше** заявленной даты и данных за дату нет → удалённая проверка силами
  нашей техподдержки (а не произвольный ответ про передачу/настройку объекта).

**Коротко:** mileage-путь — точный «станок» для расхождения пробега; общий путь —
универсальный ассистент для всего остального, более свободный и всегда с ревью
оператора.

## 3a. Разбор «общих» заявок по форматам вложений (`automate_batch`)

Заявка с ≥2 ТС (несколько вложений или список в теле) идёт в **пакетный разбор**:
строится таблица «ТС × дата × вердикт». Извлечение ТС учитывает реальные форматы
дочерних Россетей (по одному парсеру на формат):

| Парсер | Формат | Пример ПО |
|---|---|---|
| `_parse_act_blocks` | мульти-акт PDF: один «Акт №» = один ТС, своя дата/ПЛ/ГЛОНАСС | Волжское, Жигулёвское |
| `_parse_body_vehicles` | список ТС прямо в теле заявки (номер + дата + пробег) | Прихоперское, тело-списки |
| `_parse_summary_table` | сводная таблица (строка = ТС) | сводные/ОДКР письма |
| `_parse_grouping_table` | XLSX-отчёт «Группировка <дата>» | Ульяновское |

Глобальный дедуп ТС по вложениям (один номер не задваивается между актами/телом,
ключ — `_plate_dedup_key` без хвоста региона).

**Разбор нескольких ТС по ТЕЛУ письма** (сессия 14, ⏳ в работе — на ветке, не в
main): раньше в пакет уходили только заявки с ≥2 вложениями/номерами в теме. Теперь,
если в **теме + теле** насчитывается ≥2 разных номера (`_parse_body_vehicles`), заявка
идёт в пакетный разбор по объектам — каждый ТС со своей датой/пробегом из своей
строки тела, номера только из темы добираются с общей датой (65649: тема = дата, в
теле 20 номеров построчно — раньше терялись все, кроме первого).

### Формат-профили ПО — `_FORMAT_PROFILES` / `resolve_format_profile`
Мягкая привязка по подстроке `company_name` (производственное отделение). Профиль
задаёт: где искать данные (`subject`/`attachments`/оба), кардинальность (`single`/
`multi`), формат вложений (`none`/`act_per_object`/`one_doc_many`/`table`) и
человекочитаемую **подсказку для LLM-промпта**. Если совпадения нет — работает обычная
эвристика. Покрыты: Волжское, Жигулёвское, Чапаевское, Самарское, Прихоперское,
Северное, Северо-Восточное, Ульяновское ПО, Пензаэнерго, Мордовэнерго, Оренбургэнерго.

### Вердикт-по-дате
Ключ строки разбора = (номер, дата, файл): один ТС за разные даты получает разные
вердикты. Вердикт редактируется оператором (`POST /batch/verdict`) и влияет на общий
ответ (`compose_answer`).

**Ручная правка гос.номера/даты оператором** (сессия 14): если OCR исказил номер
(М567МВ→MS69MB, 64722) или ошиблась дата, оператор исправляет их — и в разборе по
объектам (правится ТОЛЬКО строка `(старый номер, дата, файл)`), и в одиночном
анализе (`plate`/`date` override в `automate`). Система **заново запрашивает гео по
исправленному номеру** и обновляет вердикт/трек/пробег. Есть пометка неверного
ФОРМАТА номера. Допустимые: Глушение / Данные верны / Не было питания /
Нет данных / Терминал подключился / Проверить. Глушение приоритетнее совпадения
пробега (спуфинг ≠ «данные верны», даже если км совпали).

## 4. Доп. факты для ИИ (обновления 2026-06)

Кроме базовых флагов, в анализ подаются (помогают «читать» данные как график):
- **`профиль_дня_по_4ч`** — напряжение/скорость/пакеты по 6 отрезкам суток.
- **`анализ_графика`** — окно движения (`движение_с/по`), время падения напряжения,
  крупные разрывы трека.
- **`ситуация_с_данными`** — при `no_data`: дата последнего сообщения vs дата
  неисправности (раньше/позже/в дату).
- **`данные_оборвались_среди_дня`** — данные были, но прервались рано (<17:00) и не
  возобновились (признак потери питания среди дня, 64275).
- **`проверка_данных_после_даты`** — возобновились ли данные позже + дата.
- **`период_анализа`** — если дата задана интервалом («15.06-16.06»), пробег/данные
  считаются за весь интервал.
- **Знания о питании/напряжении** в промпте: данные есть → питание было; нет данных →
  нет питания или связь(SIM); норма ≥12 В (легковые)/≥24 В (грузовые); >12 В без
  спутников/скорости → глушение.

## 5. Слабые места и что улучшать

- **Парсинг даты/номера хрупок** на нестандартных письмах. Частично закрыто:
  «период за», сегментация мульти-акт PDF, столбец «Дата выезда», несколько дат на
  объект, фоллбэк номера из темы. ⏳ Покрытие форматов — постоянная задача.
- ✅ **Кэш `result_cache`** — инвалидируется при новом комментарии/решении заявки.
- ✅ **Питание, пропавшее среди дня** — детектируется (`данные_оборвались_среди_дня`)
  + few-shot-эталон (64275).
- ✅ **Пороги телеметрии** — параметризуются через env (`TELEMETRY_*`).
- **Few-shot зависит от наполнения** `TrainingSample` — пополняется автоматически
  (закрытие заявки + бэкафилл из решённых) + шаблоны категории (`fetch_templates_for_category`).
  Полезно периодически догонять бэкафиллом.
- ✅ **Петля обратной связи** (`ai_feedback`): оператор оценивает разбор good/bad
  (+ тип ошибки + комментарий + правильная категория); экран «Оценки ИИ» — основа для
  точечной правки промпта/правил и отбора верных разборов в few-shot.
- ⏳ **Vision графиков** — пока числовой профиль/события; image-vision — отдельный эпик.
- ⏳ **Спецномера в списках/таблицах/вложениях** (сессия 14): `extract_all_plates`/
  `_PLATE_STD_RE` ловят только стандартный формат — спецномера спецтехники («4 цифры +
  2 буквы» / «2 буквы + 4 цифры», напр. `АА8941`, `2874СЕ`) из списков, таблиц и
  вложений НЕ извлекаются. Поэтому «общая» заявка со спецмашинами во вложении может
  распознать не все ТС (65676: 2 машины → нашёл 1). Одиночный `_is_special_vehicle`
  спецформат распознаёт, но массовое извлечение — нет.

## 6. Принципы гибкости в рамках

ИИ должен быть гибким в формулировках, но не выходить за рамки категорий каталога
и не выдумывать данных. Это достигается тем, что **решение опирается на
детерминированные факты** (флаги телеметрии, `ситуация_с_данными`,
`данные_возобновились`), а LLM лишь красиво формулирует ответ в заданных границах —
это и экономит токены, и держит ответ корректным для клиента и оператора.
