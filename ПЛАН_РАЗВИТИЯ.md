## 🎯 Единый сценарий работы ИИ (идея, к реализации)

Привести поведение к одному стилю для заявок С вложениями и БЕЗ: **ИИ разбирает →
анализирует → предлагает ответ → оператор решает.**

Этапы (единые карточки-шаги в блоке «Анализ заявки»):
1. **Разбор** — что извлечено из заявки/вложений (номера, даты, ПЛ, факт). Для
   заявок без вложений «разбор» = парсинг тела; с вложениями = таблица по объектам.
2. **Анализ** — телеметрия/флаги/график по каждому объекту (или по одному ТС).
3. **Предложение ответа** — кнопка «Составить ответ» (всегда): с вложениями — по
   таблице, без — по одиночному анализу. Редактируемый вердикт влияет на ответ.
4. **Решение оператора** — править вердикт/текст, вставить в комментарий, отправить.

Идеи реализации:
- Один общий компонент-«мастер» с 4 шагами и единым прогрессом (вместо разрозненных
  кнопок), одинаковый для обоих случаев (число объектов 1..N).
- Авто-определение режима: ≥1 извлекаемое вложение → табличный разбор; иначе —
  одиночный анализ; UI одинаковый, меняется только наполнение шага 1–2.
- Везде показывать «уверенность ИИ» и причину (reasoning), решение всегда за оператором.

## Инфраструктура — апгрейд сервера (для 2+ сотрудников + глобальный DPI)
**Проблема.** Текущий сервер `155.212.186.165`: **1 CPU-ядро, 960 МБ RAM**
(свободно ~80 МБ, постоянный своп). OCR многоактных сканов — CPU+RAM-затратный:
- OCR вынесен в поток (event loop не виснет), но 1 ядро = разбор больших PDF идёт
  медленно и конкурирует за единственное ядро при работе **двух сотрудников**.
- Память — узкое место: пиксмап страницы при 150 DPI ~19 МБ, при 300 DPI ~70-90 МБ;
  два параллельных разбора больших сканов при высоком DPI → своп/риск OOM-kill.
- Поэтому **глобальный подъём DPI (150→200/300) сейчас НЕ делаем** — на этом железе
  это риск падения сервиса по памяти, а не просто «медленнее».

**Что даёт апгрейд (цель: 2-4 ГБ RAM, 2+ ядра):**
- [ ] Параллельная работа 2+ сотрудников без деградации (разные ядра под OCR/запросы).
- [ ] Безопасный **глобальный DPI 200-300** → меньше нераспознанных гос.номеров в
  сканах (кейс 64722/64725: OCR не читал номер на 150 DPI).

## Принцип
ИИ — гибкий в формулировках, но в рамках: решение опирается на **детерминированные
факты** (флаги телеметрии, ситуация с данными, отредактированные вердикты), а LLM
формулирует. Это повышает точность, экономит токены и делает ответ предсказуемым.
