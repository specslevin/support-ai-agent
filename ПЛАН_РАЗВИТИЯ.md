## 🎯 Приоритет 2 — Массовые (сводные/ОДКР) заявки (заявка 63317)

**Контекст:** клиент присылает один отчёт на месяц по многим ТС для сверки;
большинство заявок уже выполнены. ИИ делает по каждому ТС вердикт (Глушение /
Проверить / Данные верны / Нет данных). Клиент требует ОДИН общий ответ по всем.

### 2.1. Редактируемый вердикт в таблице разбора
- [ ] В таблице разбора по объектам дать пользователю **менять вердикт** каждого
  ТС вручную (выпадающий список: Глушение / Данные верны / Не было питания /
  Нет данных / Проверить). Изменения сохраняются и используются в общем ответе.

### 2.2. Учёт «нет данных на дату, но появились позже»
- [ ] Для каждого ТС определять: данных на дату нет, но позже возобновились →
  отдельная формулировка в общем ответе («в указанную дату данных не было,
  восстановлены <дата>»).

### 2.3. Генерация одного общего ответа с группировкой по вердиктам
- [ ] Сгруппировать ТС по (отредактированным) вердиктам и сформулировать единый
  ответ, например:
  - Глушение → «по ТС … в указанный период фиксировалось воздействие средств
    подавления GPS-сигнала»;
  - Данные верны → «по ТС … сбоев не обнаружено, данные корректны»;
  - Не было питания → «по ТС … отсутствовало питание, восстановлено <дата>»;
  - Нет данных → «по ТС … требуется удалённая проверка силами техподдержки».
- [ ] Учитывать, что многие заявки уже выполнены — ответ это сверка, не повторная
  работа.

## Принцип
ИИ — гибкий в формулировках, но в рамках: решение опирается на **детерминированные
факты** (флаги телеметрии, ситуация с данными, отредактированные вердикты), а LLM
формулирует. Это повышает точность, экономит токены и делает ответ предсказуемым.
