## 🎯 Приоритет 1 — Прокачка знаний ИИ

Цель: ИИ всё точнее повторяет решения опытного оператора, реже ошибается,
экономит токены за счёт детерминированных фактов.

### 1.1. Чтение графиков и телеметрии (заявка 64281)
**Проблема:** ИИ написал «не было питания, 17.06 отсутствовало напряжение,
мин. напряжение 0 В», хотя по графику видно движение и напряжение ~27.8 В.
**Задачи:**
- [ ] Разобрать баг `min_power_v = 0` при реальных ~27.8 В — вероятно, в выборку
  напряжения попадают нулевые/пустые `pwr_ext` пакеты; чинить расчёт `min_power_v`
  и `power_off_ratio` (игнорировать заведомо невалидные нули, брать медиану/перцентиль).
- [ ] Учитывать факт движения: если ТС двигалось и есть валидное напряжение —
  не выносить вердикт «не было питания».
- [ ] (Большой блок) Дать ИИ «читать» графики: подавать в анализ агрегаты рядов
  (напряжение во времени, скорость, спутники) или vision-анализ изображения
  графика. Сначала — числовые агрегаты рядов в факты, позже — vision.

### 1.2. Автонаполнение базы эталонов (few-shot / TrainingSample)
- [ ] При закрытии заявки оператором автоматически сохранять
  (факты телеметрии → ответ оператора) как `TrainingSample` (не только вручную).
- [ ] Панель в UI для просмотра/курирования эталонов (что попадает в few-shot).
- [ ] Метрика: доля заявок, где ответ ИИ совпал с финальным ответом оператора.

### 1.3. Детект «питание пропало среди дня» (заявка 64275)
- [ ] Сигнал: данные за день есть, но резко обрываются и НЕ возобновляются в тот
  же день, а появляются позже → «не было питания, восстановлено <дата>».
  Сейчас закрыто только few-shot-эталоном; нужен детерминированный детектор
  (осторожно с порогами — регрессионный прогон по выборке).

### 1.4. Расширение парсинга и устойчивость
- [ ] Покрытие новых форматов писем/актов дочерних Россетей (даты, номера).
- [ ] Параметризация порогов телеметрии (`_POWER_OFF_V`, `_MIN_SAT`,
  `_TELEPORT_KMH` …) по типам терминалов.
- [ ] TTL/инвалидация `result_cache` (после нового комментария или догрузки данных).

## 🎯 Приоритет 2 — Массовые (сводные/ОДКР) заявки (заявка 63317)

**Контекст:** клиент присылает один отчёт на месяц по многим ТС для сверки;
большинство заявок уже выполнены. ИИ делает по каждому ТС вердикт (Глушение /
Проверить / Данные верны / Нет данных). Клиент требует ОДИН общий ответ по всем.

### 2.1. Редактируемый вердикт в таблице разбора
- [ ] В таблице разбора по объектам дать пользователю **менять вердикт** каждого
  ТС вручную (выпадающий список: Глушение / Данные верны / Не было питания /
  Нет данных / Проверить). Изменения сохраняются и используются в общем ответе.

### 2.2. Учёт «нет данных на дату, но появились позже»
- [ ] Для каждого ТС определять: данных на дату нет, но позже возобновились →
  отдельная формулировка в общем ответе («в указанную дату данных не было,
  восстановлены <дата>»).

### 2.3. Генерация одного общего ответа с группировкой по вердиктам
- [ ] Сгруппировать ТС по (отредактированным) вердиктам и сформулировать единый
  ответ, например:
  - Глушение → «по ТС … в указанный период фиксировалось воздействие средств
    подавления GPS-сигнала»;
  - Данные верны → «по ТС … сбоев не обнаружено, данные корректны»;
  - Не было питания → «по ТС … отсутствовало питание, восстановлено <дата>»;
  - Нет данных → «по ТС … требуется удалённая проверка силами техподдержки».
- [ ] Учитывать, что многие заявки уже выполнены — ответ это сверка, не повторная
  работа.

## Принцип
ИИ — гибкий в формулировках, но в рамках: решение опирается на **детерминированные
факты** (флаги телеметрии, ситуация с данными, отредактированные вердикты), а LLM
формулирует. Это повышает точность, экономит токены и делает ответ предсказуемым.
